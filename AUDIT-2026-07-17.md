## 1. CRITICAL — money / trust / legal (do first)

### C1. Portal booking trusts the browser for price, and self-signup accounts are active instantly
`api.js` (public API) reprices server-side — correct. The in-app portal path does **not**: sell/cost are computed in the browser (`db.js:709-728`), `ship.js:189` books on the platform master FedEx account, and the browser writes the shipment record (with whatever `sell`/`cost` it likes) into its own namespace. `db.js requestAccess` (436-497) creates an **immediately active** account with a session token, no approval. A stranger can self-register and book real FedEx labels on your account at a price they choose (or raw cost).
**Fix:** reprice + stamp sell/cost server-side at booking (mirror `api.js`); make self-signup accounts `status:"pending"` until an admin activates. **Real money on every occurrence — top of the list.**

### C2. Fake bookings escape demo mode — fabricated tracking pushed to Shopify + emailed to buyers
When live booking isn't enabled (`canBook=false` for any real account with live-rates off), "Print label" on Ship (`7763`), Batch (`10345`), Autopilot (`12369`), and the order modal (`9219`) still call `onShipped()` with a **random** tracking number (`newTracking`, `337`). `onShipped` (`6867`) marks the order fulfilled, **pushes the fake tracking to the live Shopify store**, pushes to other connectors, and emails the buyer "your order shipped." A real merchant with Shopify connected but live booking off sends fake tracking to real customers.
**Fix:** gate the simulated path to `currentUser.demo===true` only; otherwise show an explicit "live booking not enabled" error and never call `onShipped`.

### C3. Privacy policy states two things that are false today
- `public/privacy.html:32` promises you honor Shopify data-deletion webhooks; `shopify-compliance.js:57-75` only `console.log`s and returns 200 — deletes nothing. And order PII **is** stored server-side (`db.js`), contradicting the handler's own comment.
- `privacy.html:31` says store tokens are "stored encrypted"; they're plaintext JSON in Supabase (`shopify-auth.js:86-87` → `settings.shopifyConns` → synced as plain jsonb).
**Fix:** implement real redaction (or a tracked manual workflow) + encrypt tokens at rest, OR correct the policy immediately. A false statement in a live privacy policy is the highest-risk legal item. *(Attorney should see this.)*

### C4. Unauthenticated SSRF via connector `storeUrl`
`woocommerce.js`, `magento.js`, `netsuite.js`, `salesforce.js`, `quickbooks*.js` fetch a user-supplied `storeUrl` server-side with **no auth gate** and no private-host check, reflecting response snippets back. Anyone can use these as an outbound request proxy against internal hosts (e.g. cloud metadata endpoints).
**Fix:** add the `scAuth` session gate (same one every other function uses) + `isPrivateHost`/https validation to all connectors.

*Legal items C3 plus the following are attorney-review flags, not legal advice.*

### C5. "Insurance" should be "declared value" everywhere customer-facing
UI says "Insurance $" / "Insure $" / "Auto-insure" (`9850, 8023, 13543`), but your own Terms §7 say you're not an insurer. Selling "insurance" unlicensed is a regulated-activity flag in several states. Mechanical rename of display strings (not state keys) → big risk reduction.

### C6. Platform markup shown under FedEx's exact published fee names
Each surcharge is marked up (`2600-2617`) then displayed as "Fuel Surcharge," "Delivery Area Surcharge," etc. — carrier-official names on amounts that include your margin. FTC junk-fee/mislabeling exposure. The API docs go further (`public/api-docs.html:39`: "what the API quotes is what you're billed" on "FedEx contracted pricing") and contradict your own Terms.
**Fix:** a one-line disclosure on the breakdown ("resale prices; itemized fees may include our margin"), and rewrite the API-docs pricing line.

### C7. Retail footer claims a carrier "partner agreement," contradicting your Terms
`6104`: "shipping is provided under partner carrier agreements" vs Terms §2 "not affiliated with… FedEx, DHL, USPS, UPS." The Freightwire landing already has the correct non-affiliation wording (`5966`) — copy it to the retail footer. Also review the "Save 10–15% by switching to FedEx" claim (`6014`) for substantiation.

## 2. HIGH — broken features & security holes

### H1. "See costs & spend" (seeCosts) not enforced on Orders and Shipments
Both components receive `showMoney` and never use it — rates/totals/prices always render, even when an admin turned the flag off (Dashboard/Batch/Reports do honor it). *(Dashboard "Spend" was fixed earlier this session; Orders + Shipments still leak.)*
**Fix:** gate the est-rate/total/price cells on `showMoney` like Batch does.

### H2. Shopify Checkout Rates + Product-Catalog "Pull from Shopify" read connection keys nothing writes
`10781` and `11157` read `settings.conn.shopify` / `settings.shopifyShop` — but Shopify connections live in `settings.shopifyConns`. Result: the entire live-checkout-rates feature (a headline Pro feature, and the whole `shopify-rates.js` backend) and Shopify product import always fail "Connect your Shopify store first."
**Fix:** use `shopifyConnFor(settings)` in both places. (Small, high-value.)

### H3. `email.js` is a low-cost open relay
After any valid token (trivially obtained via self-signup), the caller controls recipient + 100KB of arbitrary HTML sent from `notify@shippingcloud.net`, no rate limit. Phishing / domain-reputation risk. **Fix:** restrict to templated sends, add per-account limits, require verified accounts.

### H4. Webhook SSRF filter bypassable + login has no brute-force protection
`isPrivateHost` (`api.js:114`) only string-matches dotted-quad IPv4 — decimal/hex/IPv4-mapped-IPv6 encodings pass. And `db.js login` has no rate limit/lockout (signup and 2FA do). **Fix:** resolve + range-check the actual IP; add per-IP/per-account login throttle.

### H5. Rate limits are per-warm-container (soft), so paid-call amplification isn't bounded
FedEx quote/label and Anthropic assistant calls are limited by in-memory counters that reset on cold start and don't share across concurrent Lambdas. **Fix:** move counters to a shared store for the money endpoints.

## 3. MEDIUM — misleading shells & incomplete features

- **QuickBooks (Online + Desktop), Salesforce, NetSuite are connect-only shells.** Server actions exist; **no client code calls them.** Users complete OAuth ("Connected ✓") and nothing ever posts. QBO tokens die in an hour (never refreshed). **Fix:** hide or label "setup only — invoicing coming soon."
- **Billing "Prepaid balance / Add funds" is fiction.** `addFunds` increments a number; no payment is collected and nothing ever deducts. A customer can "add $500" and believe they're funded. **Fix:** gate/remove until payments exist.
- **Local-mode Login "Create account" discards the request** (`3103`) — sets a "Request received" note, writes nothing. **Fix:** route through `requestAccess`.
- **The one live automated email omits its tracking number** (`6908` doesn't pass `tracking`), and the `customer@example.com` fallback actually gets emailed. **Fix:** pass `tracking`/`trackUrl`; never send to the fallback.
- **"Bring your own carrier accounts" and "Platform carrier accounts" toggles control nothing** (their UI was removed; flags still toggle). **Fix:** drop from the admin catalogs.
- **UPS integration is half-removed:** `ups.js` (test + rate) is never called; `settings.ups`/`runUpsTest` are unreachable dead client state. *(This is what I mis-stated earlier — the file exists but is fully orphaned; no live UPS anywhere.)* **Fix:** delete, or restore the UI when you wire UPS for real.
- **Login-scratch wipe misses half the ship-form keys** (`5425`) — a leftover Saturday/signature/third-party-bill-to flag from a prior session silently re-prices the first quote of a new session. **Fix:** add the missing keys to `SCRATCH_KEYS`.
- **Subscription plans advertise limits/features that don't exist** ("25/500/2,500 labels/mo" enforced nowhere; "Branded tracking page" doesn't exist). **Fix:** trim to real features or enforce the limits.
- **Returns is a dead-end tab** — logs an RMA, tells you to book the label on Ship "then paste tracking here," but there's no input to paste into, no label booking, no status/delete. Biggest tier-expectation gap for a shipping platform. **Fix:** book a real FedEx `RETURN` label from the RMA (the ship.js path supports it).
- **Legal fix-soon:** outdated carrier names — "DHL SmartMail Parcel" (`2181`), "SmartPost" (`2213`), USPS "Commercial Plus" (`2897`) all renamed years ago (trademark + accuracy); the nightly full-tenant PII backup is **emailed as an attachment** (`guardian-nightly.mjs`) — keep it in Drive only; Google Fonts loaded remotely from the landing page (GDPR flag) — self-host (both fonts are OFL); a hand-built FedEx wordmark replica exists in code (`891-914`, currently unused) — delete it; two divergent privacy policies are live — pick one canonical.
