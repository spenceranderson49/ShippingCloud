## 🔴 FIRST — the wiped rates (Lagence + other clients)

**What happened:** a fresh or customer browser tab loads with the built‑in *default* rate table for a split second before your real rates finish downloading from the cloud. Under the old code that momentary empty default could get saved back up and **overwrite** the real negotiated rates. It was worst for clients whose rates live in per‑client assignments + profile services/surcharges (not the base cost grid) — which is exactly Lagence.

**Get Lagence's rates back right now (they are not gone):**
1. Admin portal → **Backups & Restore** (left sidebar, the ↺ icon).
2. Find the `rateRules` snapshot from **before** the wipe (each row shows a date + entry count + a sample so you can confirm it's the full one).
3. Click **Restore**. Your current version is snapshotted first, so it's reversible. The page reloads onto the recovered data.
4. Close any *other* admin tabs first so they can't re‑overwrite it.

**Why it can't happen again — four independent guards now stand between a stray tab and your rates:**
| Layer | Guard |
|---|---|
| Client (mount) | A component's *first‑render* value is **never** pushed to the cloud. Only genuine later edits sync. |
| Client (global stores) | Rates/users/clients writes are **held** until the first full cloud load finishes; anything edited in that window is re‑queued by a recovery pass, so nothing is lost. |
| Server (F20, hardened) | A write that would replace real rate content — measured across **assignments + every profile's services/surcharges + cost tables**, not just base costs — with an empty default is **refused** and snapshotted. |
| Server (all critical stores) | Every overwrite of rates/users/clients snapshots the current value first (newest 10 kept). Total wipes and severe shrinks are refused outright. |

Your box catalog got the same treatment (an empty/seed `boxes` write can no longer erase a populated catalog).

Remaining real gaps (honest list, ranked):
1. **Pick‑list workflow objects aren't linked** (biggest one) — details in the next section.
2. **Returns don't restock automatically.** A return on the shipping side doesn't yet add the item back into WMS on‑hand. Worth wiring next.
3. **Addresses / ship‑from locations vs. WMS warehouses** are two lists. A WMS "warehouse" (with bins) and a shipping "ship‑from address" describe the same physical place but aren't linked, so you maintain both.
4. **POS sales draw down stock, but shipping‑side manual stock adjustments and WMS cycle counts are the only way back up** — there's no single "inventory ledger" view that shows every reason a number moved (sold, shipped, received, counted, returned). A unified ledger would make the number trustworthy.

None of these are on fire. #1 and #2 are the two I'd do next.

## Pick lists & packing slips — how they coordinate, and how to tie them together

- **Shipping side:** the "Pick list" and "Packing slip" buttons on Orders/Batches are **stateless print helpers** — they aggregate whatever's in the current batch and print, using your shared logo/title settings. Nothing is tracked afterward.
- **Warehouse side:** a WMS pick list is a **tracked workflow object** — created, assigned, scanned item‑by‑item (scan‑to‑pick), then pack‑verified. It has a status and a history.

So today they **share the paper** (same document look) but not the **process** (only WMS tracks who picked what, when). That's why it feels disconnected.

**To tie it together (recommended):** when WMS mode is on, make the shipping‑side "Pick list" button *create a WMS pick‑list object* instead of just printing — so a batch queued for shipping shows up as a real, trackable pick task, and pack‑verify closes the loop back to the label. That turns two parallel tools into one pipeline: **batch on the shipping side → tracked pick/pack in the warehouse → label prints → stock decrements.** This is a focused next project (not done tonight — it changes a live workflow and I didn't want to touch that on the same night rates were at risk).

## Dropoffs & Mailboxes — what they are, and whether you need them

Both are **retail pack‑and‑ship counter** tools (think a UPS‑Store‑style front desk), added for EliteWorks parity:

- **Dropoffs** = an intake log for packages a customer physically hands you to ship. You log it, then move it *pending → shipped*. Useful only if you (or your clients) take walk‑in packages over a counter.
- **Mail Boxes** = a rental registry — who rents which physical mailbox, their contact, and status (*active / vacant / overdue*). Useful only if you rent out mailboxes.

**Do you need them?** If your business is **3PL / ecommerce fulfillment**, no — they're clutter for that audience. If any of your clients run a **retail shipping storefront**, they're exactly right. My suggestion: keep them behind a "Front‑desk tools" toggle that's **off by default**, so they only appear for the counter‑type clients who want them. Right now they sit under a "Front desk" group in the warehouse nav — fine to leave, but I'd gate them.
