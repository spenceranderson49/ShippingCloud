ShippingCloud API

API Reference — v1

Integrate your systems directly: live discounted FedEx rates on your own pricing, label booking, shipments, pickups, and billing. On ShipHub sites this same API is the ShipHub API — identical endpoints. Support: support@shippingcloud.net

Base URL & authentication

https://<your-platform-domain>/api/v1

Every request needs your API key (issued by your account manager — keys start with sck_live_). Send it either way:

Authorization: Bearer sck_live_xxxxxxxxxxxxxxxx
# or
X-API-Key: sck_live_xxxxxxxxxxxxxxxx

All requests and responses are JSON. Errors return a proper HTTP status and {"error":{"code","message"}}. Rate limit: 240 requests/min per key (429 beyond that). Prices are quoted live from FedEx on your account's contracted pricing — what the API quotes is what you're billed.

POST/v1/rates — quote a shipment

curl -s https://<domain>/api/v1/rates \
  -H "Authorization: Bearer $KEY" -H "Content-Type: application/json" \
  -d '{
    "from_zip": "84101",
    "to_zip": "30301",
    "residential": true,
    "signature": "none",
    "packages": [{ "weight": 3, "length": 12, "width": 9, "height": 4, "declared_value": 250 }]
  }'
FieldNotes
from_zip, to_zipRequired. 5-digit US ZIPs (international: add to_country).
packages[]Required. weight (lb), length/width/height (in), optional declared_value ($ per package — coverage is priced into the quote).
residentialDefault true. Controls Home Delivery vs Ground.
signaturenone | indirect | direct | adult — priced into the quote.
saturday_deliveryBoolean, Express services only.
package_typeOptional FedEx packaging (e.g. fedex_small_box) — unlocks One Rate flat pricing where the shipment qualifies.

Response: rates[] sorted cheapest-first, each with service_code, service, amount, delivery_days. Quote the exact shipment you'll book — dimensional weight and accessorials are priced per request.

POST/v1/labels — book a label

curl -s https://<domain>/api/v1/labels \
  -H "Authorization: Bearer $KEY" -H "Content-Type: application/json" \
  -d '{
    "service_code": "ground",
    "from": { "name": "Warehouse A", "address1": "215 S State St", "city": "Salt Lake City", "state": "UT", "zip": "84101", "phone": "8015550100" },
    "to":   { "name": "Jane Doe", "address1": "127 Market St", "city": "Atlanta", "state": "GA", "zip": "30301", "phone": "4045550142" },
    "packages": [{ "weight": 3, "length": 12, "width": 9, "height": 4 }],
    "reference": "SO-1042"
  }'

Extra fields: invoice_number, po_number, department (print on the label as FedEx references), signature, saturday_delivery, declared_value_total, ship_date (YYYY-MM-DD), label_stock (4x6 default).

Response 201: label_id, tracking_number, tracking_url, charge.amount (what you're billed), and label_pdf_base64 — decode and print. The service is re-quoted at booking so the charge always matches a live rate.

POST/v1/labels/batch — up to 100 labels at once

{ "shipments": [ { /* a normal /v1/labels body */ }, { … } ] }

Returns 207 with a results[] array in the same order — each entry is either a booked label (status:"success") or status:"error" with its index, so one bad shipment never fails the rest.

POST/v1/returns — prepaid return label

Same body as /v1/labels. By default from/to are swapped (the parcel ships from the buyer back to you); pass "swap": false if you already set them return-direction. Response adds is_return: true.

GET/v1/shipments · POST/v1/shipments/{label_id}/void

GET /v1/shipments?limit=50 lists your labels newest-first with status and charge. Void marks the label refunded-in-progress; unshipped labels refund automatically.

GET/v1/tracking/{tracking_number}

Returns the label's current platform status plus a carrier tracking_url for full scan history.

POST/v1/pickups — schedule a carrier pickup

{ "date": "2026-07-14", "ready_time": "09:00", "close_time": "17:00",
  "total_weight": 12, "package_location": "FRONT",
  "address": { "address1": "215 S State St", "city": "Salt Lake City", "state": "UT", "zip": "84101" } }

GET/v1/services · GET/v1/rate-card

/v1/services lists every service enabled on your account (the service_code values for booking). /v1/rate-card returns your account's pricing configuration machine-readably — how each service and accessorial is priced for you.

GET/v1/billing/summary · GET/v1/invoices

/v1/billing/summary?month=2026-07 → shipment count + total charges for the month. /v1/invoices?month=2026-07 → full line items (add &format=csv for a spreadsheet-ready download).

When your account manager has issued invoices, GET /v1/invoices lists them (id, number, status, balance) and GET /v1/invoices/{id} returns one with full line items and paid/balance — your system can reconcile payments automatically.

POST/v1/addresses/validate

FedEx-backed validation + residential/commercial classification: send {"address1","city","state","zip"}, get valid, classification, and a normalized address to correct typos before you buy a label.

GET/v1/labels/{label_id} — re-download

Lost the response? Recent label PDFs are stored and re-downloadable by label_id.

Idempotency — safe retries

Idempotency-Key: order-1042-attempt   # any unique string, per label

Send it on POST /v1/labels. If your request times out and you retry with the same key, you get the original booking back (marked Idempotency-Replayed: true) — a retry can never double-book or double-charge.

Webhooks — signed push events

POST /v1/webhooks   { "url": "https://yourapp.com/hooks/shipping", "events": ["label.created","label.voided"] }

Registration returns a whsec_… secret (shown once). Every delivery is signed — verify X-SC-Signature as hex HMAC-SHA256 of the raw body with your secret. GET /v1/webhooks lists, DELETE /v1/webhooks/{id} removes. Max 5 per account; deliveries are best-effort — poll /v1/shipments as the source of truth.

International shipments

Set to.country (ISO-2 or full name) and include customs: { "contents": "Cotton t-shirts", "value_total": 240 } on POST /v1/labels — the customs value is required for a truthful declaration; per-package declared_value works too.

GET/v1/pickups · cancel · GET/v1/account

GET /v1/pickups lists scheduled pickups; POST /v1/pickups/{confirmation}/cancel cancels one. GET /v1/account returns your account name, key prefix, enabled-service count and limits — handy as a connectivity check.

Errors

StatuscodeMeaning
401missing_key / invalid_keyNo key, or a bad/revoked one.
403service_not_enabled / account_inactiveAsk your account manager.
422invalid_request / service_unavailableFix the request; re-check /v1/rates.
429rate_limited240 req/min per key.
5xxrates_unavailable / booking_failedTransient carrier issue — retry with backoff.

Migrating from another shipping API

This API replaces legacy shipping APIs one-for-one for rating and booking: POST /v1/rates ↔ rate quotes, POST /v1/labels ↔ order booking + label retrieval (no polling — the label PDF returns in the booking response), /v1/shipments ↔ shipment lookup, plus billing endpoints those APIs never had. Field names are plain snake_case; nothing about the upstream provider leaks into your integration.
